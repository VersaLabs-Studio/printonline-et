### C2: Delivery Fee Flow — Iteration 1

| # | Action | Result | Notes |
|---|--------|--------|-------|
| 1 | Add item → goto `/checkout` | ❌ FAIL | **Bug #1:** Delivery fee always shows "Free" regardless of sub-city or cart total |

**User Report (C2-I1):**
> "The delivery fee says free although the order is totaled less than 5000 birr. I picked the subcity Lemi Kura and the delivery fee is still says free. I even cycled through subcity changes and delivery fee stays free."

**Root Cause:** No sub-city selector existed in checkout step 1. `deliveryInfo.subCity` was only set once from profile fetch (and only if null). `getDeliveryFee()` returned 0 when subCity was null because the calculator returned 0 for unknown zones. `OrderSummaryDetails` incorrectly showed "Free delivery" for any 0 fee.

**Fix Commit (C2-I1):**
1. Added sub-city dropdown to `OrderProfileSection` (step 1)
2. Created `handleSubCityChange` callback syncing to `CartContext.deliveryInfo`
3. Created `PUT /api/account/profile` API endpoint
4. Added inline profile editor in checkout step 1 (Edit/Save/Cancel toggle)
5. Fields: Name, Phone, TIN, Company, Address 1/2, Sub-City, Woreda
6. Save persists to Supabase `customer_profiles` (same source as `/account`)
7. Live delivery fee preview updates on sub-city change
8. `OrderSummaryDetails` now only shows "Free delivery" when cart >= 5000 ETB
9. `OrderReviewStep` shows actual calculated fee, not hardcoded "Free/Included"

**Files Modified:**
- `components/order/OrderProfileSection.tsx` (rewrite)
- `components/order/OrderSummaryDetails.tsx` (added subCity, zone info)
- `components/order/OrderReviewStep.tsx` (added deliveryFee, cartTotal props)
- `app/order-summary/page.tsx` (added handleSubCityChange, handleProfileUpdate, new prop pass-through)
- `app/api/account/profile/route.ts` (NEW — PUT endpoint)

### C2: Delivery Fee Flow — Iteration 2

| # | Action | Result | Notes |
|---|--------|--------|-------|
| 1 | Select Yeka sub-city in step 1 | ❌ FAIL | **Bug #2:** Delivery preference card shows **70 ETB**, order summary shows **28 ETB** |

**User Report (C2-I2):**
> "Under the delivery preferences button between the two, it provides the correct delivery fee of 70 birr, but says differently here on the order summary section."

**Root Cause:** The delivery preference card showed the raw `baseFee` (70 ETB) while the order summary used `calculateDeliveryFee()` which applied the quantity discount matrix (50 items = 0.4× multiplier = 28 ETB). This was a UI display mismatch, not a calculation error.

**The Math:**
| Factor | Value |
|--------|-------|
| Yeka base fee | 70 ETB |
| Cart quantity | 50 items |
| Quantity tier | 21–50 → 0.4× |
| Discount | 42 ETB |
| Final fee | 28 ETB |

**Fix Commit (C2-I2):**
1. Updated live preview in `OrderProfileSection` to apply `getQuantityMultiplier(cartCount)`
2. Added breakdown line: `Base 70 ETB · -42 ETB bulk discount (50 items)`
3. Added breakdown line in `OrderSummaryDetails` when quantity discount applies

**Files Modified:**
- `components/order/OrderProfileSection.tsx`
- `components/order/OrderSummaryDetails.tsx`
- `app/order-summary/page.tsx` (pass cartCount)

### Order Status Workflow Refactor

**User Request:**
> "Implement this order status flow: Order Confirmed -> Design Under Review -> On Hold (if issue detected) -> Approved for Production -> Printing in Progress -> Ready for Delivery -> Out for Delivery -> Delivered. Apply it across account page, CMS workflow, order confirmation, and order detail pages."

**Status Mapping (old → new):**
| Old | New |
|-----|-----|
| `pending` | `pending` (pre-payment, unchanged) |
| `confirmed` | `order_confirmed` |
| `design_review` | `design_under_review` |
| `on_hold` | `on_hold` (unchanged) |
| `approved` | `approved_for_production` |
| `printing` | `printing_in_progress` |
| `ready` | `ready_for_delivery` |
| `out_for_delivery` | `out_for_delivery` (unchanged) |
| `delivered` | `delivered` (unchanged) |
| `cancelled` | `cancelled` (unchanged) |

**CMS Transition Rules Updated:**
```
pending → [order_confirmed, cancelled]
order_confirmed → [design_under_review, on_hold, cancelled]
design_under_review → [on_hold, approved_for_production, cancelled]
on_hold → [design_under_review, cancelled]
approved_for_production → [printing_in_progress, cancelled]
printing_in_progress → [ready_for_delivery, cancelled]
ready_for_delivery → [out_for_delivery, delivered, cancelled]
out_for_delivery → [delivered, cancelled]
delivered → [] (terminal)
cancelled → [] (terminal)
```
