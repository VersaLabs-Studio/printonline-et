### C2: Delivery Fee Flow — Iteration 1

| # | Action | Result | Notes |
|---|--------|--------|-------|
| 1 | Add item -> goto `/checkout` | FAIL | **Bug #1:** Delivery fee always shows "Free" regardless of sub-city or cart total |

**User Report (C2-I1):**
> "The delivery fee says free although the order is totaled less than 5000 birr. I picked the subcity Lemi Kura and the delivery fee is still says free. I even cycled through subcity changes and delivery fee stays free."

**Fix Commit (C2-I1):**
1. Added sub-city dropdown to `OrderProfileSection` (step 1)
2. Created `handleSubCityChange` callback syncing to `CartContext.deliveryInfo`
3. Created `PUT /api/account/profile` API endpoint
4. Added inline profile editor in checkout step 1 (Edit/Save/Cancel toggle)
5. Fields: Name, Phone, TIN, Company, Address 1/2, Sub-City, Woreda
6. Save persists to Supabase `customer_profiles` (same source as `/account`)
7. Live delivery fee preview updates on sub-city change
8. `OrderSummaryDetails` now only shows "Free delivery" when cart >= 5000 ETB
9. `OrderReviewStep` shows actual calculated fee, not hardcoded "Free/Included"

**Files Modified:**
- `components/order/OrderProfileSection.tsx` (rewrite)
- `components/order/OrderSummaryDetails.tsx` (added subCity, zone info)
- `components/order/OrderReviewStep.tsx` (added deliveryFee, cartTotal props)

### Bug: Messages Sending Failed (Empty Error `{}`)

**Root Cause:** The `messages` table RLS policies used `auth.uid()`, which is Supabase Auth's user ID. This app uses better-auth, not Supabase Auth. `auth.uid()` returns `NULL` for every user, so RLS silently blocked all inserts, selects, and updates.

**Fix:**
1. Updated `001_create_messages_table.sql` — Replaced broken `auth.uid()` policies with permissive policies (`USING (true)`), since access is gated through authenticated Next.js API routes
2. Created `POST /api/messages` and `GET /api/messages` authenticated API endpoints
3. Created `POST /api/messages/mark-read` endpoint
4. Updated `lib/supabase/messages.ts` — `sendMessage`, `getMessagesByOrder`, `markOrderMessagesAsRead` now call the authenticated API routes
5. Added better error logging and toast messages

## Session 4: 2026-04-28 — Message Layout + Read Receipts Fix

### Issue 1: Message Layout Alignment — FIXED

**Root Cause:** Previous 3 iterations used `flex justify-end`/`justify-start` on the message row, but ancestor containers (`h-[400px] overflow-y-auto`, `flex-1 overflow-y-auto -mx-2`) and the inner `flex` wrapper weren't guaranteeing full-width context for justify-content to distribute space properly.

**Fix Applied:**
1. **Replaced `justify-end`/`justify-start` with `ml-auto`/`mr-auto`** on the inner message container. `margin: auto` is resolved after flex item sizing and works even when ancestor constraints prevent `justify-content` distribution.
2. **Added `w-full min-w-0`** to the root `space-y-1` wrapper in `MessageList.tsx`.
3. **Added `w-full`** to scroll containers in both `MessagePortal.tsx` and `messages/[orderId]/page.tsx`.

**Files Modified:**
- `components/chat/MessageList.tsx` — `ml-auto`/`mr-auto` on message rows; `w-full min-w-0` on root
- `components/chat/MessagePortal.tsx` — `w-full` on scroll container
- `app/(account)/messages/[orderId]/page.tsx` — `w-full` on scroll container

**Fallback documented:** If `ml-auto` still fails in user's browser, switch to CSS Grid with `justify-self-end`.

### Issue 2: Read Receipts Inconsistent — FIXED

**Root Cause:**
1. **Optimistic local state update** after `markOrderMessagesAsRead` could get out of sync with DB if the API partially failed or if realtime UPDATE events were missed.
2. **Realtime UPDATE events not firing reliably** for bulk updates (multiple rows updated at once).
3. **No fallback** for missed realtime events when tab was backgrounded or connection dropped.

**Fix Applied:**
1. **Replaced optimistic update with refetch-after-mark-read.** After `markOrderMessagesAsRead` succeeds, the client calls `getMessagesByOrder()` again to get the true server state.
2. **Added 30s polling fallback.** A `setInterval` refetches messages every 30 seconds to catch any missed realtime updates.
3. **Added window focus refetch.** When the user returns to the tab, messages are refetched and marked as read again.
4. **API returns `updatedCount`.** `POST /api/messages/mark-read` now returns `{ success: true, updatedCount: N }` so the client can verify the operation actually affected rows.
5. **Used `useCallback`** for `loadMessages` to satisfy `react-hooks/exhaustive-deps` and prevent stale closures.

**Files Modified:**
- `components/chat/MessagePortal.tsx` — Refactored message loading with `useCallback`, polling, focus refetch
- `app/(account)/messages/[orderId]/page.tsx` — Same refetch pattern
- `lib/supabase/messages.ts` — `markOrderMessagesAsRead` now returns `updatedCount`
- `app/api/messages/mark-read/route.ts` — Returns `updatedCount` from `.select("id")`
