# PrintOnline Ethiopia — Version 3.8 Summary

**Date:** June 2026

---

## What's New in This Update

Version 3.8 is a **reliability and polish** release. The customer-facing experience is unchanged, but the underlying email infrastructure has been migrated to a professional provider, all email templates have been unified under a premium design, and several production bugs have been fixed.

---

### 1. Email System Migrated to Resend

We migrated our **outgoing (transactional) email infrastructure** from the legacy EthioTel SMTP relay to **Resend**, a modern transactional email API.

**Why this matters:**

- **Reliable delivery** — Resend's API has proper deliverability, DKIM, SPF, and DMARC signing configured at the domain level, so verification, password reset, and order emails consistently reach the inbox (not spam).
- **Speed** — Sub-second API delivery vs. variable SMTP relay times.
- **Observability** — Bounce, complaint, and delivery status are tracked per-message.
- **No more silent failures** — Failed sends surface immediately with proper error logs.

**What routes through Resend now:**

- Account verification emails
- Password reset emails
- Welcome emails (new account)
- Order confirmation emails
- Order status update emails
- Payment failure notifications
- Account deletion confirmations
- Admin new-order notifications

**What did NOT change:** Customer-facing email addresses (`order@printonline.et`) remain the same. The "from" address displayed in customers' inboxes is unchanged.

---

### 2. Email Infrastructure — Important Note

> **PLEASE READ:** Our **sending** infrastructure is now Resend (reliable), but our **receiving** mailbox (`order@printonline.et`) is still hosted with EthioTel webmail.

During testing, we observed **intermittent delays and occasional email loss on EthioTel's webmail**. This is an infrastructure issue on EthioTel's side and is **outside of our control** — we have escalated it, but we cannot resolve it from the application layer.

**What this means for you (the team):**

- ✅ **Outgoing emails are 100% reliable.** All customer-facing emails (verification, password reset, order confirmations) are sent via Resend. Customers will always receive them.
- ⚠️ **Incoming customer emails to `order@printonline.et` may be delayed or occasionally lost at the EthioTel webmail layer.** We recommend checking the inbox at least twice daily and following up on missing customer replies by phone if needed.
- 📞 **For time-sensitive customer communication, also call** the phone number on the contact page.

This is a transitional state. Migrating the receiving mailbox to a more reliable provider is on the roadmap for a future release.

### 3. Google OAuth — Fully Functional

Single sign-on with Google is now fully wired up. Customers can register and sign in with one click using their Google account.

**Setup required (one-time, by the team):**

The following redirect URIs must be registered in Google Cloud Console for the OAuth client to work in production:

```
https://www.printonline.et/api/auth/callback/google
https://printonline.et/api/auth/callback/google
```

(Both are required because `www.printonline.et` redirects to the non-www version at the edge.)

If customers see a `redirect_uri_mismatch` error after clicking "Sign in with Google," the URIs above need to be added to the Google Cloud Console under:
**APIs & Services → Credentials → OAuth 2.0 Client IDs → Authorized redirect URIs**

### 5. Email Verification Flow — Fixed

After verifying their email, customers are now correctly redirected to their **account page** (`/account`) instead of the homepage with a confusing `?error=token_expired` query string.

**The fix:** The signup form now passes an explicit `callbackURL: "/account"` to the authentication client, telling it where to send the user after the verification token is consumed. Previously, an empty/default callback URL caused the verification link to point at `/`, and any re-hit of the same link (already consumed once) surfaced the `token_expired` error on the homepage.

**Customer experience:** Click verification link → account is verified → automatically signed in → redirected to `/account`. No more "something went wrong" pages.

### 6. Contact Form — Correct Inbox

The contact page email display was pointing at `design@printonline.et`, which is **not a configured inbox**. It has been corrected to `order@printonline.et`, which is the team inbox monitored for general inquiries.

**Fixed:** The displayed email on `/contact` now matches the inbox that actually receives the messages.

### 7. Production Hardening — Origin Guard

Added a **deployment-time safety check** that prevents the app from building in production if `BETTER_AUTH_URL` is misconfigured to point at `localhost`, `127.0.0.1`, or `0.0.0.0`.

**Why this matters:** A misconfigured `BETTER_AUTH_URL` in Vercel's environment variables causes every authentication request to silently fail with a `403 "Invalid Origin"` error — but only at runtime, on the first user signup attempt. Now, the problem surfaces immediately at **build/deploy time** with a clear error message explaining exactly which env var to fix.

This is a defense-in-depth layer for the deployment checklist documented in `AGENTS.md`.
