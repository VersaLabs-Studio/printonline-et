## 1. Executive Summary

v3.7 builds on the v3.6 CMS infrastructure to make the **storefront fully dynamic** and the **auth system production-ready**. Every hardcoded homepage section (hero slideshow, deals, testimonials) becomes CMS-configurable. The auth flow gets OAuth integration (Google, Facebook, TikTok), real email verification, and password recovery. A cookie/privacy compliance banner is added. Product metadata gets manual quantity entry and interval configuration. The CMS save UX is refined to "save and stay" instead of "save and redirect."

  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,

  WHERE is_active = true;

  FOR ALL USING (true) WITH CHECK (true);

### F14: Image Upload to site-images Bucket

**Current state:** HeroSlideForm, DealForm, and TestimonialForm use text-only URL inputs for `image_url` / `avatar_url` fields. No file upload capability for v3.7 features.

**Target:** Each form component gets a file uploader that:
1. Sends files to the existing `POST /api/upload/site-images` endpoint
2. Uses folder names: `hero` (hero slides), `deals` (homepage deals), `testimonials` (testimonial avatars)
3. Auto-fills the `image_url` field in the form with the returned public URL from Supabase Storage
4. Shows a live preview of the uploaded image
5. Supports both file upload AND manual URL text input (the URL field stays, upload is additional)

**Implementation:**

**14.1 HeroSlideForm** — Add file upload:
- Add an "Upload Image" button alongside the URL input
- On file select: POST to `/api/upload/site-images` with `folder: "hero"`
- On success: auto-fill the `image_url` field with the returned `url`
- Show live preview using `form.watch("image_url")` (already present)
- Keep the text URL input for external URLs

**14.2 DealForm** — Add file upload:
- Same pattern, POST to `/api/upload/site-images` with `folder: "deals"`
- Auto-fill `image_url` field
- Keep URL text input

**14.3 TestimonialForm** — Add file upload for avatar:
- Same pattern, POST to `/api/upload/site-images` with `folder: "testimonials"`
- Auto-fill `avatar_url` field
- Keep URL text input

**14.4 Migration Seed Updates:**
- Update `025_homepage_hero_slides.sql` seed INSERT to use relative paths from `/public` directory (no change needed — existing images at `/sample1.jpg` etc. are in public)
- Update `026_homepage_deals.sql` seed to use `/public` images for default deal images
- Update `027_homepage_testimonials.sql` seed avatars to use `/public` images or leave as placeholder paths

**Upload Flow:**
1. Admin clicks "Upload Image" button in form
2. File picker opens (accept="image/*", max 10MB)
3. File uploaded to `/api/upload/site-images` as FormData with `folder` field
4. API stores file in Supabase Storage `site-images` bucket at `{folder}/{timestamp}-{random}-{name}.{ext}`
5. API returns `{ success, url, path }` where `url` is the public URL
6. Form's `image_url` field is set to `json.url`
7. Live preview updates automatically via `watch` reactivity

**Files to modify:**
- `components/cms/settings/HeroSlideForm.tsx` — Add file upload button + upload handler
- `components/cms/settings/DealForm.tsx` — Add file upload button + upload handler
- `components/cms/settings/TestimonialForm.tsx` — Add file upload for avatar

**Files to create:** None (reuses existing `site-images` upload infrastructure)

## 14. Risk Mitigation

| Risk | Mitigation |
|------|------------|
| OAuth env vars not available | Graceful fallback: hide social buttons if env vars missing. Email/password always works. |
| Homepage breaks if DB tables not migrated | Every dynamic section has hardcoded fallback data |
| Image reorder data loss | Optimistic update + error rollback. Re-fetch on failure. |
| Cookie banner blocks content | Use `position: fixed; bottom: 0` with proper z-index. Dismissible with localStorage. |
| better-auth OAuth callback issues | Test each provider individually. Use `callbackURL` param. |
| Double-toast on save | Remove duplicate toast from form `onSubmit` handlers |
| Email delivery failure | Graceful error handling. Log failures. Don't block registration on email send. |
| Image upload exceeds 10MB or unsupported type | API returns 413/400. Form shows toast error. File input accepts only images via `accept="image/*"`. |
| site-images bucket doesn't exist in Supabase | App creates bucket on first upload via `supabaseAdmin.storage.createBucket()` if needed. Fallback: text URL input still works. |

## 17. Notes & Constraints

1. **better-auth OAuth is free** — no paid services required. Google/Facebook/TikTok developer accounts are free to create.
2. **No breaking changes** — All homepage sections have hardcoded fallbacks. Product quantity changes are additive (nullable columns with defaults).
3. **Schema consistency** — New tables follow the exact same pattern as v3.6 tables (UUID PK, timestamps, RLS, triggers).
4. **Hook pattern consistency** — All new hooks follow the TanStack Query pattern from v3.6 (query key, stale time, mutation with invalidation).
5. **CMS pattern consistency** — All new CMS pages use `CMSPageHeader`, form/dialog patterns from v3.6.
6. **Image upload** — Reuses existing `/api/upload/site-images` endpoint for all new image uploads with folder names: `hero`, `deals`, `testimonials`. Each form component (HeroSlideForm, DealForm, TestimonialForm) gets an upload button alongside the text URL input. Files are stored in the `site-images` Supabase Storage bucket (public, no RLS needed for reads).
7. **Seed data** — Migration seed INSERTs use files from the `/public` directory (e.g., `/sample1.jpg`). After migrations are applied, CMS admins should replace these with uploaded images via the CMS forms. The seed provides working fallback data.
8. **The plan does NOT include** the Eventology blueprints mentioned in the user's message — those are for a separate project.
