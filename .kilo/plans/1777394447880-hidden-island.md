## Bug 1: Stale Delivery Fee (Root cause of Bugs 2 & 3)

### Root Cause

In `OrderProfileSection.tsx` (lines 354-363), the delivery method `RadioGroup.onValueChange` calls **two separate context updaters** in sequence:

```tsx
onValueChange={(val) => {
  setDeliveryMethod(val);        // calls setDeliveryInfo({...deliveryInfo, deliveryMethod: 'pickup'})
  ...
  onSubCityChange("Bole");       // calls setDeliveryInfo({...deliveryInfo, subCity: 'Bole'})
}}
```

Both calls spread the **same stale `deliveryInfo`** from the render closure. The second call **overwrites** the first — so `deliveryMethod` reverts to its old value while `subCity` updates. Result: when user selects "pickup", the context stores `{deliveryMethod: 'home', subCity: 'Bole'}`, which calculates as Bole zone fee (200 ETB) instead of 0.

The `OrderProfileSection`'s local `liveDeliveryFee` calculation is correct (it uses local state), but the sidebar `OrderSummaryDetails` uses `getDeliveryFee()` from context which is stale.

### Fix

**File: `components/order/OrderProfileSection.tsx`**

Replace the `onValueChange` handler to use a **single `setDeliveryInfo` call** that combines both the method and sub-city changes:

```tsx
onValueChange={(val) => {
  setDeliveryMethod(val);
  if (val === "pickup") {
    // Single atomic update: both method and sub-city
    onSubCityChange(null);  // pickup doesn't need sub-city
  } else if (val === "other") {
    onSubCityChange(altSubCity || null);
  } else {
    onSubCityChange(watchedSubCity || null);
  }
}}
```

**File: `context/CartContext.tsx`**

Refactor `setDeliveryInfo` to accept a **partial updater** pattern so callers can update individual fields without stale spread:

```tsx
setDeliveryInfo: (info: Partial<DeliveryInfo> | ((prev: DeliveryInfo) => DeliveryInfo)) => void;
```

Then in CartContext:
```tsx
const setDeliveryInfo = useCallback((infoOrFn: Partial<DeliveryInfo> | ((prev: DeliveryInfo) => DeliveryInfo)) => {
  setDeliveryInfoState(prev => {
    const update = typeof infoOrFn === 'function' ? infoOrFn(prev) : { ...prev, ...infoOrFn };
    return update;
  });
}, []);
```

Update all callers to pass partial objects instead of spreading stale state.

**File: `app/order-summary/page.tsx`**

Update `handleDeliveryMethodChange`, `handleSubCityChange`, `handleAlternateAddressChange` to use partial updater pattern instead of `{...deliveryInfo, ...}`.

### Files to Modify

| File | Change |
|------|--------|
| `context/CartContext.tsx` | Refactor `setDeliveryInfo` to accept partial updater |
| `app/order-summary/page.tsx` | Update all delivery handlers to use partial updater |

## Bug 2: Pickup Shows ETB 200 Fee

### Root Cause

Same as Bug 1 — the stale closure causes `deliveryMethod` to remain `'home'` after selecting pickup, so `getDeliveryFee()` calculates Bole zone fee instead of 0.

### Fix

Resolved by Bug 1 fix. Additionally, the `CartContext.getDeliveryFee` should validate: if `deliveryInfo.deliveryMethod === 'pickup'`, always return 0 regardless of subCity.

**File: `context/CartContext.tsx`** — In `getDeliveryFee`:
```tsx
if (deliveryInfo.deliveryMethod === 'pickup') return 0;
```

This is already handled by the calculator, but adding a guard in the context ensures no stale subCity can leak through.

## Bug 3: Cart Page Subtotal Shows Plain Subtotal

### Current Behavior

`CartSummary.tsx` (line 75-82) shows `subtotal` (products only) and `total` (subtotal + delivery). But there's no separate "Delivery" line, so the user sees a "Subtotal" that doesn't match the final total.

### Fix

**File: `components/cart/CartSummary.tsx`**

Add a delivery fee line between subtotal and total. If delivery is 0 (pickup or free delivery), show "Calculated at checkout" or the delivery method label. Since the cart page doesn't have delivery preferences set yet, show a contextual message:

```tsx
<div className="flex justify-between items-center">
  <span className="...">Subtotal</span>
  <PriceDisplay amount={subtotal} />
</div>
<div className="flex justify-between items-center">
  <span className="...">Delivery</span>
  <span className="...">Calculated at checkout</span>
</div>
```

And update the total to show just the subtotal (since delivery isn't known yet), OR show `subtotal + delivery` if delivery is set.

Actually, per user request: "display a subtotal with delivery fee added instead of the plain subtotal since the delivery fee details are to be persisted on the next flow page" — the user wants the cart page to show `subtotal + delivery` as the displayed total. Since delivery may not be set yet on the cart page, the approach should be:

1. If `delivery > 0` (delivery info already set in context), show: Subtotal, Delivery, Total
2. If `delivery === 0` (no delivery info or pickup), show just: Subtotal (which equals total)

**File: `components/cart/CartSummary.tsx`** — Add delivery line when `delivery > 0`.

## Bug 5: Print Receipt Button Doesn't Work

### Root Cause

`ConfirmationDetails.tsx` line 324: `onClick={() => window.print()}` — This triggers the browser's print dialog for the entire page, not the Chapa receipt.

The Chapa receipt URL (`https://checkout.chapa.co/checkout/test-payment-receipt/...`) is only available during the Chapa redirect (< 3 seconds) and cannot be accessed afterward.

However, the `payment_receipt` column in the DB stores the full Chapa verification response (from `verify` API or webhook), which contains: `tx_ref`, `amount`, `currency`, `method`, `status`, `created_at`.

### Fix

**File: `components/order/ConfirmationDetails.tsx`**

Replace `window.print()` with a proper receipt print action:

1. Create a hidden printable receipt div with all payment details from `payment_receipt`
2. Use `window.print()` with a print-specific CSS media query that only shows the receipt section
3. OR generate a downloadable PDF receipt

The simplest approach: Create a receipt modal/overlay with proper formatting, then call `window.print()` on it. Use CSS `@media print` to hide everything else.

**Also:** Add a link to the Chapa receipt if the `tx_ref` is available. Chapa receipt URLs follow the pattern: `https://checkout.chapa.co/checkout/test-payment-receipt/{tx_ref}` (test mode) or similar for production. We can't guarantee this URL format, but we can try.

Actually, the simplest fix is to improve the print functionality:
1. Add `@media print` CSS to hide non-receipt elements
2. Format the receipt section for printing
3. Add a "Download Receipt" button that generates a clean printable view

## Bug 6: Payment Verification Gets Stuck on "Verifying Payment..."

### Root Cause

In `order-confirmation/page.tsx`, the verification `useQuery` (lines 35-60):

```tsx
const { data: verificationData, isLoading: isVerifying } = useQuery({
  queryKey: ["verify-payment", txRef],
  queryFn: async () => {
    const res = await fetch(`/api/payments/verify?tx_ref=${txRef}`);
    const data = await res.json();
    if (data.success) {
      clearCart();
      await fetch("/api/send-order-email", { ... });  // ← AWAITED
    }
    return data;
  },
  enabled: !!txRef,
  retry: 1,
});
```

Multiple issues:

1. **Email sending is awaited inside `queryFn`**: If `/api/send-order-email` is slow (sends 2 emails — customer + admin), the `queryFn` won't return until both emails send. This can take 5-10+ seconds, during which the user sees "Verifying Payment...".

2. **No timeout**: If Chapa's `verify` API is slow (test mode can be slow), the verification hangs indefinitely.

3. **Race condition with webhook**: The webhook may have already confirmed the order. The verify API does check for `payment_status === "paid"` and returns immediately, but if the webhook hasn't arrived yet, it calls `chapa.verify()` which is an external API call.

4. **`retry: 1`** means it retries once on failure, adding more delay.

### Fix

**File: `app/order-confirmation/page.tsx`**

1. **Don't await the email call** — fire-and-forget:
   ```tsx
   if (data.success) {
     clearCart();
     fetch("/api/send-order-email", { ... }).catch(() => {});  // Fire and forget
   }
   ```

2. **Add a timeout** to the verification query (15 seconds):
   ```tsx
   const controller = new AbortController();
   const timeout = setTimeout(() => controller.abort(), 15000);
   const res = await fetch(`/api/payments/verify?tx_ref=${txRef}`, { signal: controller.signal });
   clearTimeout(timeout);
   ```

3. **Add polling fallback**: If verification fails or times out, poll the order status every 5 seconds (the webhook may confirm it):
   ```tsx
   refetchInterval: (query) => {
     if (query.state.data?.success) return false; // Stop polling on success
     if (query.state.errorCount >= 3) return false; // Stop after 3 failures
     return 5000; // Poll every 5s
   },
   ```

4. **Better loading UI**: Show elapsed time and a "We're verifying with Chapa" message with a manual "Check Again" button.

**File: `app/api/payments/verify/route.ts`**

The API already has the idempotency check (returns immediately if already paid). No changes needed here.
